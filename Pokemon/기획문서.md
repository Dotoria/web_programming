# Poketest 기획 문서

> 작성일: 2026-07-11
> 대상: `web_programming/Pokemon` 프로젝트 (JSP/Tomcat 기반 "포켓몬 유형 테스트" 웹앱)

---

## 1. 프로젝트 개요

**Poketest**는 MBTI류 성격 테스트처럼, 사용자가 여러 문항(5단계 리커트 척도: 매우 그렇다 ~ 전혀 그렇지 않다)에 답하면 18개 포켓몬 타입 점수를 합산해 **"당신의 포켓몬 타입 조합"**을 결과로 보여주는 재미용 웹앱입니다.

- 기술 스택: JSP + Servlet(Tomcat 10.1), 순수 JS/CSS, `question.csv` 기반 문항 데이터
- 개발 목적: 웹프로그래밍 학습용 개인 프로젝트 (`poketest.html`에 "웹프로그래밍 초보가 제작"이라 명시)
- 현재 상태: **초기 프로토타입.** 페이지 뼈대와 일부 스크립트만 존재하고, 핵심 기능(채점·결과·리뷰 저장)은 미구현

## 2. 현재 상태 분석

### 2.1 파일별 현황

| 파일 | 상태 | 비고 |
|---|---|---|
| `poketest.html` | ✅ 거의 완성 | 랜딩 페이지. 소개, 주의사항, 리뷰 폼. `시작하기` → `test.html` |
| `test.html` | 🟡 뼈대만 존재 | `question.csv`를 동기 XHR로 읽어 **문항 전체를 한 페이지 테이블로** 렌더링. "다음으로" 버튼이 `test.html`로 자기 자신을 새로고침 — 다음 문항 이동/제출 로직 없음. 상단 "processor bar 삽입 예정" 주석 → 원래는 문항별 진행형 UI를 의도했던 것으로 보임 (현재 구현과 설계 의도 불일치) |
| `question.csv` | 🔴 거의 비어있음 | 52줄 중 **2줄만 채워짐**. 값 인코딩도 행마다 다름 (`fire0~fire4` vs `fly1~fly5`) — 시작 인덱스 불일치, 타입명도 `fly` vs `type.js`의 `flying`으로 불일치 |
| `type.js` | 🔴 껍데기만 존재 | 18개 타입 카운터 변수 선언만 있고, 이 변수들을 채점에 사용하는 로직이 전혀 없음 |
| `review.jsp` | 🔴 미착수 | "jsp 작업 페이지"라는 플레이스홀더 텍스트만 있음. `poketest.html`의 리뷰 폼은 `action`이 주석 처리되어 있어 **어디에도 제출되지 않음** |
| `review.js` | 🟡 버그 있음 | `document.getElementsByName("name")`은 NodeList를 반환하므로 `.value` 접근 시 `undefined` 에러. 정규식 `/^\.{1, 20}$/`도 의도(20자 이하 검증)와 다르게 동작(이스케이프된 `.` + 공백 포함 잘못된 quantifier) |
| `poketest.jsp`, `test`(index.jsp) | 🔴 미사용/잔재 | `index.jsp`는 Tomcat 설치 기본 환영 페이지 그대로. `poketest.jsp`는 사실상 빈 파일 |
| `WEB-INF/web.xml` | 🔴 미설정 | Tomcat 기본 `web.xml` 그대로. `welcome-file-list`가 `poketest.html`을 가리키지 않고, 서블릿 매핑도 전혀 없음 |
| `tomcat.css/svg/txt`, `bg-*.png`, `favicon.ico`, `RELEASE-NOTES.txt`, `asf-logo-wide.svg` | ⚪ 잔재 | Tomcat 설치 시 기본 제공되는 샘플 웹앱 파일들. 실제 서비스에는 불필요 |
| `testInfo.txt` | ⚪ 참고자료 | MBTI/에니어그램/Big5 등 성격유형 프레임워크 정리 메모. 코드는 아니고 문항·결과 설계용 리서치 노트로 추정 |

### 2.2 핵심 문제 요약

1. **채점 엔진 부재**: 답변을 타입 점수로 환산하는 로직이 어디에도 없음 (`type.js` 변수 선언뿐).
2. **결과 페이지 부재**: 테스트를 끝까지 풀어도 "당신의 타입"을 보여줄 화면 자체가 없음 — 이 앱의 존재 이유인 핵심 기능이 빠져있음.
3. **문항 데이터 부족**: 52문항 슬롯 중 2개만 작성, 그마저 인코딩 규칙이 통일 안 됨.
4. **테스트 진행 흐름 미완성**: 한 페이지에 전체 문항을 몰아서 렌더링(설계 의도와 다름), 페이지 이동/제출 버튼이 실제로 동작하지 않음.
5. **리뷰 기능 백엔드 없음**: 폼은 있지만 서버로 전송되지 않고, `review.jsp`도 로직이 없음. 저장소(DB/파일)도 미정.
6. **배포 설정 안 됨**: `web.xml`이 기본값이라 `poketest.html`이 진입점으로 동작 안 함 (현재는 Tomcat 기본 페이지가 뜸).
7. **Tomcat 샘플 잔재**: 실제 서비스 시 정리 필요.
8. **버전 관리 없음**: 현재 git 저장소가 아님 — 이후 기능이 늘어날수록 위험.

## 3. 목표 아키텍처 제안

지금 단계에서 스택을 갈아엎기보다, **현재 JSP/Servlet + Tomcat 구조를 유지**하며 빈 부분을 채우는 것을 권장합니다 (학습 목적 프로젝트이고, 이미 Servlet 학습 흐름에 맞춰져 있음).

- **문항/채점**: `question.csv` → 서버(Servlet) 또는 클라이언트(JS)에서 파싱 → 사용자가 선택한 값들을 타입별로 합산
- **결과 산출**: 점수가 가장 높은 상위 1~2개 타입을 조합해 보여주는 방식 (포켓몬처럼 "불꽃/비행" 듀얼 타입 콘셉트)
- **리뷰 저장**: 학습 목적이면 CSV/텍스트 파일 저장도 충분하지만, Servlet + DB(H2 또는 MySQL) 조합으로 가면 실습 범위가 자연스럽게 확장됨
- **세션 관리**: 문항별 페이지 이동 시 사용자의 응답을 유지하려면 `HttpSession` 또는 클라이언트 sessionStorage 필요

## 6. 결정이 필요한 오픈 이슈

- 문항 수 및 타입당 배분(균등 배분 vs 특정 타입 가중치)
- 결과에서 단일 타입만 보여줄지, 포켓몬처럼 듀얼 타입 조합을 보여줄지

- 진행 UI를 "문항 하나씩 + progress bar"로 갈지, 지금처럼 "여러 문항 한 페이지"로 유지할지
